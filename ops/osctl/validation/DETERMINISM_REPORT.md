## Summary

The OSCTL minimal ledger core is **deterministic** under fixed input:

| Property | Status | Mechanism |
|----------|--------|-----------|
| Identical input → identical output | PASS | `canonical_dumps()` + pure fold/render |
| Stable key ordering | PASS | Recursive `_sort_keys()` + `json.dumps(sort_keys=True)` |
| Canonical timestamps | PASS | Schema enforces `YYYY-MM-DDTHH:MM:SS[.mmm]Z` |
| No random values | PASS | No `random`, `uuid`, `time.time`, or `datetime.now` in serialize path |
| Replay consistency | PASS | Same fingerprint across repeated fold/render |

## Serialization

**Module:** `ops/osctl/core/serialize.py`

```python
# Canonical line properties:
# - UTF-8
# - Keys sorted lexicographically (recursive)
# - ensure_ascii=True
# - separators=(",", ":")
# - No trailing newline in dumps (newline added only on append)
```

**Evidence:**

- Permutation test: dict key order `{z, a, list[{y,x}]}` → identical bytes across 5 runs.
- Sample hash (deploy fixture): `7aa4c1e2837b7d13…` (SHA-256 prefix of canonical event).

**Timestamp contract:**

- Events MUST supply `ts` at ingest time (caller-controlled, not generated by core).
- Validator rejects non-UTC ISO8601 (`TS_PATTERN` in `schema/events.py`).
- Core does not mutate or normalize timestamps on append — **determinism depends on caller supplying canonical `ts`.**

## Non-Determinism Boundaries (Explicit)

These are **outside** core guarantees:

| Source | Owner | Impact |
|--------|-------|--------|
| `ts` at append time | Human / CI caller | Ledger content varies if caller uses wall clock |
| Manual projection edits | Operator | `verify` detects drift vs replay |
| Concurrent append without lock | Operator | Filesystem race — not solved in Phase 1 |
| Git merge conflicts on JSONL | Operator | Human resolution required |

Core guarantees apply when: **ledger bytes are fixed** and **projections are regenerated via `project`**.
