## 3. Decision Options and Consequences

### H1 — Canonical Ledger Authority

**Question:** Should canonical ledger authority live at `ops/state/ledger/events.jsonl` or `ops/osctl/ledger/events.jsonl`?

#### Option A — `ops/state/ledger/events.jsonl` (Option L)

| Dimension | Impact |
|-----------|--------|
| **Architectural impact** | Aligns code defaults (after LR-2) with `ARCHITECTURE_FREEZE` F-001, `LEDGER_MODEL.md`, `GOVERNANCE.md`. Separates **operational truth plane** from **kernel package tree**. |
| **Replay impact** | Single append target; replay input unambiguous. `FREEZE_v1` fixture ledger demoted to validation-only. |
| **Operational impact** | Operators append and verify at `ops/state/`. Rituals and human playbooks naturally reference operational tree. |
| **Staging impact** | Stage **one** ledger path in anchor commit. Demote or relocate `ops/osctl/ledger/` before treating anchor as canonical. |
| **Governance complexity** | Low — docs already declare this path; LR-2 updates `paths.py` only. No freeze bump required if treated as implementation alignment. |
| **Long-term maintainability** | High — operational data lives outside kernel; kernel upgrades do not co-mingle with live bytes. |
| **Determinism risk** | Low — one write path; verify gates drift. |
| **Agent confusion risk** | Low after LR-2 — CLI default matches freeze declaration. Until LR-2, agents may append to wrong path if they follow `paths.py`. |

#### Option B — `ops/osctl/ledger/events.jsonl` (Option O)

| Dimension | Impact |
|-----------|--------|
| **Architectural impact** | Keeps ledger adjacent to kernel and validation fixtures. Requires amending F-001, `LEDGER_MODEL.md`, `GOVERNANCE.md`, `README.md` and bumping freeze to `osctl-freeze/1.5.1`. |
| **Replay impact** | Single path after `ops/state/ledger/` removed or archived. Validation fingerprint in `FREEZE_v1` remains aligned with default. |
| **Operational impact** | Operators work inside kernel tree for live truth — blurs tooling vs operations mentally. |
| **Staging impact** | Must **not** stage both ledgers. Delete or archive `ops/state/ledger/` after doc amendment. |
| **Governance complexity** | **High** — freeze text change, multiple doc edits, freeze bump protocol. |
| **Long-term maintainability** | Medium — coupling live operational bytes to kernel directory increases accidental fixture/live confusion. |
| **Determinism risk** | Low once single path enforced. |
| **Agent confusion risk** | Medium — "master" naming in root docs and `ops/state/README.md` still describe state tree as operational home until rewritten. |

#### Option C — Dual ledger (mirror both paths)

| Dimension | Impact |
|-----------|--------|
| **Architectural impact** | **Violates** ADR-001, P-001, P-003 in `INVARIANT_REGISTRY.md`. Preserves split-brain indefinitely. |
| **Replay impact** | **Catastrophic** — next append to one path only; replay divergence guaranteed. |
| **Operational impact** | Operators cannot know which file is truth without external convention. |
| **Staging impact** | Anchor commit would encode **two authorities** — unreconcilable. |
| **Governance complexity** | Maximum — perpetual reconciliation burden. |
| **Long-term maintainability** | **Unmaintainable** — drift is a matter of when, not if. |
| **Determinism risk** | **Critical** — identical ledgers today; byte divergence on first divergent append. |
| **Agent confusion risk** | **Critical** — agents will read/write arbitrary path. |

**Status today:** Option C de facto (both exist, byte-identical). **Must be collapsed to A or B.**
