# OSCTL Human Commit Sequence

**Mode:** Human commit planning only  
**Rule:** No staging, commit, push, merge, or deploy is authorized by this file.  
**Commit shape:** Small, reversible, one authority domain per commit.

---

**Purpose:** Apply approved governance entrypoint cleanup and redirects before deeper reductions.

Allowed paths:

- `ops/osctl/README.md`
- `ops/osctl/GOVERNANCE.md`
- `ops/osctl/BOUNDARIES.md`
- `ops/osctl/HUMAN_BOUNDARIES.md`
- `ops/osctl/TRUST_MODEL.md`
- Root governance context only if explicitly included in approved LR-1 scope

Forbidden paths:

- `backend/**`
- `.github/**`
- `package.json`
- `package-lock.json`
- `pnpm-lock.yaml`
- `yarn.lock`
- `docker-compose.yml`
- Deploy, Railway, Cloudflare, secret, runtime, and production files

Validation command before commit:

```powershell
python ops/osctl/validation/run_validation.py
```

Rollback note: revert this commit if entrypoint navigation points to the wrong canonical source or mixes product/runtime changes.

## Sequence Rules

1. Review `git diff --stat` before every commit.
2. Review full diff for the commit scope before staging.
3. Run validation before every commit.
4. Never use `--no-verify` on trust/governance commits.
5. Do not amend after push.
6. Prefer `git revert <commit>` for rollback.
7. Stop on any mixed backend, CI, package, deploy, infrastructure, or runtime change.
